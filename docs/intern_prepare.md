# Go 后端实习面试复盘材料

## 简历可直接使用版

- 负责内容社区后端的混合微服务化改造，基于 `Go + go-zero + Spring Boot` 搭建 `gateway-api / user-rpc / interaction-rpc / search-rpc / comment-rpc / notification-rpc / sync-sidecar` 等核心服务，形成“Go 网关 + Go 领域服务 + Java 业务核心”的渐进式双栈架构。
- 设计统一网关入口，完成 JWT 鉴权、Cookie / Authorization 双模式登录态兼容、用户上下文透传、Go RPC 调用与 Java HTTP 代理聚合，统一收敛前后端响应结构，降低混合架构下的接口使用复杂度。
- 基于 `Redis` 构建互动读模型，承接点赞、收藏、评分等高频用户态状态查询，减少页面详情和列表场景下对 MongoDB 的直接回源压力，支撑高频读场景的低延迟访问。
- 基于 `RabbitMQ` 重构异步事件链路，将帖子索引同步、评论副作用、用户资料冗余同步等操作从主业务链路中解耦，采用“主写成功 + 异步消费 + 最终一致性”的方式替代重型分布式事务。
- 设计并实现 `sync-sidecar`，消费业务事件完成 `MongoDB -> Elasticsearch` 的增量同步与手动全量重建，打通发帖、审核通过、资料更新到搜索可见的完整链路，提升搜索索引修复和回补能力。
- 拆分并落地 `comment-rpc / notification-rpc`，将评论创建、删除、一级评论列表、子评论分页、通知未读数、通知列表、批量已读等能力从 Java 服务中抽离，建立更清晰的领域边界与联调接口。
- 修复评论与通知链路中的多类一致性问题，包括评论创建成功但 MQ 发布失败导致副作用缺失、二级评论删除导致 `replyCount` 重复扣减、一级评论软删除仍误发 delete 事件导致 `commentCount` 异常、状态型通知唯一索引与历史数据冲突等问题，提升系统健壮性。
- 完善搜索能力建设，基于 `Elasticsearch` 实现帖子检索、建议词、搜索历史与热度排序，并优化建议词策略和搜索结果排序逻辑，使搜索服务从基础检索能力升级为面向产品体验的独立读模型服务。
- 建设本地可观测性闭环，为 Go 服务接入 `OpenTelemetry`，为 Java 服务补充 Jaeger 导出支持，并接入 `Loki + Promtail + Grafana + Prometheus`，统一日志中的 `service / traceId / requestId / routingKey` 字段，支持 HTTP、gRPC、MQ 三类链路的联合排障。
- 负责项目文档与面试材料整理，补充根 README、服务级 README、待办清单和项目复盘文档，将系统架构、模块边界、技术选型和核心 trade-off 沉淀为可展示、可复盘的工程化材料。

## 简历项目经历写法

### 版本一：偏工程实践

使用 `Go + go-zero + Spring Boot` 设计并实现内容社区混合微服务后端，将评论、通知、搜索、互动等高频模块从 Java 业务核心中逐步拆分，结合 `Redis` 读模型、`RabbitMQ` 事件驱动同步和 `Elasticsearch` 搜索索引，实现高频读性能优化与最终一致性落地。

### 版本二：偏结果导向

负责内容社区后端微服务化改造，基于 `go-zero` 新增 `gateway-api / user-rpc / interaction-rpc / search-rpc / comment-rpc / notification-rpc / sync-sidecar` 等服务，解决了 Java 单服务中高频读、搜索同步和跨服务观测薄弱的问题，形成了可运行的双栈微服务架构与本地观测闭环。

### 版本三：偏 Go 实习方向

## STAR 版项目复盘

### S: Situation

原始系统以 Java 服务为主，承担帖子主写、管理后台和大部分业务逻辑，但随着功能增加，开始暴露几个问题：

- 高频互动和搜索场景逐渐变成性能敏感点
- 评论、通知等能力边界不够清晰，服务职责偏重
- Mongo、Redis、ES、MQ 之间的异步链路缺乏统一治理
- 分散容器日志导致排障效率较低

### T: Task

我的目标不是推倒重来，而是在已有系统上完成渐进式演进：

- 用 Go 拆出更适合独立演进的高频领域服务
- 保持外部接口统一，降低前端联调成本
- 用 MQ 和 sidecar 打通主库与搜索库的同步链路
- 补齐日志、trace、统一响应结构和开发文档

### A: Action

#### 1. 完成网关与核心 Go 服务拆分

我基于 `go-zero` 实现了以下服务：

- `gateway-api`
- `user-rpc`
- `interaction-rpc`
- `search-rpc`
- `comment-rpc`
- `notification-rpc`
- `sync-sidecar`

其中：

- 网关负责统一鉴权、聚合和 Java 代理
- Go RPC 负责高频、边界清晰的领域逻辑
- Java 继续承接帖子主写和后台功能

#### 2. 设计高频互动读模型

针对点赞、收藏、评分等高频场景，我没有把所有状态判断都放到 Mongo 查询，而是使用 Redis 做互动读模型，降低页面详情和列表查询时的回源成本。

#### 3. 通过 RabbitMQ 做异步解耦

我把帖子索引同步、评论副作用、用户资料冗余同步等操作转到 MQ 异步链路处理，并引入 `sync-sidecar` 消费事件，把 Mongo 数据同步到 ES。

#### 4. 修复评论与通知链路的一致性问题

在评论与通知服务拆分过程中，我处理了一轮比较关键的一致性问题，包括：

- 评论创建时 MQ 发布失败但接口错误返回成功
- 二级评论删除时 `replyCount` 重复扣减
- 一级评论软删除仍误发 delete 事件导致 `commentCount` 异常
- 通知状态型去重索引与历史数据不兼容

#### 5. 建设本地可观测性闭环

我为 Go 服务接入 `OpenTelemetry`，为 Java 服务补充 Jaeger 导出方式，并接入 `Loki + Promtail + Grafana`，统一日志字段中的 `service / traceId / requestId / routingKey`，使 HTTP、gRPC、MQ 三类链路能够串联排障。

### R: Result

最终我得到的是一套可以完整运行的混合微服务后端，而不是零散的服务 Demo：

- 评论、通知、搜索、互动等模块具备独立服务边界
- 网关能够统一接入 Go RPC 与 Java 服务
- 发帖 -> MQ -> ES、评论 -> 通知等链路具备最终一致性
- 本地环境支持日志检索、链路追踪和接口回归
- 项目具备较强的技术展示价值，适合用于 Go 后端实习面试

## 模块划分怎么讲

面试时不要按文件夹背诵，而是按“系统分层”来讲。

### 1. 网关层

`gateway-api` 负责统一入口、JWT 鉴权、上下文透传、Go RPC 调用、Java HTTP 代理和数据聚合。

可讲点：

- 为什么它不是单纯 Nginx 替代品，而是偏 BFF 的应用层网关
- 为什么要统一 Go 与 Java 的外部 API 形态

### 2. 领域服务层

Go 服务按高频业务拆分：

- `user-rpc`：账号与资料
- `interaction-rpc`：点赞/收藏/评分
- `search-rpc`：搜索与建议词
- `comment-rpc`：评论创建、删除、树形查询
- `notification-rpc`：通知中心

可讲点：

- 为什么这些模块适合先迁到 Go
- 为什么它们比“帖子主写”更适合优先拆出

### 3. Java 核心业务层

`platform-java` 继续承担帖子主写、管理后台和部分复杂业务聚合逻辑。

可讲点：

- 为什么不追求一次性重写
- 为什么演进式拆分比全量迁移更贴近真实工程

### 4. 异步同步层

`sync-sidecar` 负责消费 MQ 事件并维护搜索索引、日志旁路等能力。

可讲点：

- 为什么 sidecar 适合承担 ES 同步这类旁路逻辑
- 为什么搜索索引更新不应该阻塞主业务请求

## 核心架构亮点怎么讲

### 1. 渐进式微服务改造

不是从零搭建，而是在已有 Java 系统上做演进式拆分。

你可以这样回答：

> 这个项目最大的特点不是“服务很多”，而是它展示了如何在已有业务核心不下线的情况下，把适合独立演进的高频模块逐步迁到 Go。这样既能保住原系统可用性，又能让新模块按更轻的方式发展。

### 2. 多存储协同

你可以这样回答：

> 我不是把所有数据放进同一个数据库，而是根据访问模式做划分。MySQL 存用户强事务数据，Mongo 存内容和交互明细，Redis 存读模型和会话控制，ES 负责搜索。核心思想是按场景选存储，而不是追求单库统一。

### 3. 最终一致性而不是分布式事务

你可以这样回答：

> 发帖成功后搜索索引不需要强一致进入主事务，否则链路会被 ES 和下游副作用拖慢。我用 RabbitMQ 把搜索同步、通知落库和冗余信息更新移出主链路，用异步消费和重试接受短时间不一致，这是内容社区更实际的方案。

### 4. 网关聚合模式

你可以这样回答：

> 网关不是纯转发，而是负责鉴权、上下文透传和结果聚合。比如搜索页、帖子详情页需要把 ES、评论、互动状态和用户资料统一拼成一个前端可消费的结构，所以我把它设计成 BFF 风格。

### 5. 可观测性建设

你可以这样回答：

> 我专门把 `traceId`、`requestId`、`routingKey` 这些字段统一起来，再结合 Jaeger 和 Loki，把 HTTP、gRPC、MQ 三种链路串起来。这样调试评论、通知、搜索同步这类跨服务问题时，不需要靠肉眼翻多个容器日志。

## 面试高频问题与回答思路

### Q1：为什么要用 Go 和 Java 混合，而不是统一技术栈？

回答思路：

- Java 侧已有可运行业务核心，直接推翻重写风险高
- Go 更适合拆出去承接高频、边界清晰、性能敏感的服务
- 这种双栈方案更贴近真实工程里的演进，而不是理想化重构

精简回答：

> 我不是为了炫技做双栈，而是因为项目本身处于演进阶段。Java 已经承载主业务，Go 则负责新拆出的高频服务，这样可以在控制风险的前提下逐步完成微服务化。

### Q2：为什么搜索同步不直接在主业务里写 ES？

回答思路：

- ES 写入属于副作用，不应该阻塞主业务
- 搜索允许短时间延迟可见
- MQ 积压和重试比主事务失败更可控

精简回答：

> 我把 Mongo 作为主数据源，把 ES 作为搜索读模型。发帖后只要主数据写成功即可，ES 通过 MQ 异步同步，这样能降低主链路耗时，也避免 ES 抖动影响核心业务。

### Q3：为什么互动状态要用 Redis 读模型？

回答思路：

- 点赞/收藏状态是高频读，不适合每次都回源 Mongo
- 用户态状态判断需要低延迟
- Redis 更适合承接“是否已点赞/收藏”这类读优化

精简回答：

> 互动场景的问题不是写一次，而是会被页面反复读取。我用 Redis 承担用户态互动状态查询，把它和持久化明细解耦，这样详情页和列表页都能更快返回结果。

### Q4：为什么评论和通知要独立拆成 RPC？

回答思路：

- 这两个模块边界天然清晰
- 与帖子主写不同，它们更适合独立扩展和独立排障
- 拆出去后能更好地建立接口、索引、事件和联调边界

精简回答：

> 评论和通知是非常典型的独立领域。一方面调用频率高，另一方面容易形成独立的数据模型和副作用链路，所以很适合作为 Go 服务拆出来。

### Q5：你在这个项目里处理过什么比较真实的 bug？

建议答法：

- 评论创建成功但 MQ 失败，导致副作用缺失
- 二级评论删除导致 `replyCount` 双重扣减
- 旧 Mongo 索引与新服务唯一索引策略冲突
- Jaeger 只显示 Java 服务，最后定位为 Go OTLP 导出协议和端口配置问题

精简回答：

> 我印象最深的是评论链路一致性问题。评论写入成功但 MQ 发布失败时，客户端会以为成功了，但帖子计数和通知都不会更新。后来我把发布失败显式上抛，并补了写入回滚，才把这条链路补完整。

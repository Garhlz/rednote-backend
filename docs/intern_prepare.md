## 简历描述（可直接使用）

- 负责内容社区后端的混合微服务化改造，基于 `Go + go-zero + Spring Boot` 搭建 `gateway-api / user-rpc / interaction-rpc / search-rpc / comment-rpc / notification-rpc / sync-sidecar` 等核心服务，形成"Go 网关 + Go 领域服务 + Java 业务核心"的渐进式双栈架构。
- 设计统一网关入口，完成 JWT 鉴权、Cookie / Authorization 双模式登录态兼容、用户上下文透传，并通过独立封装的 `javaproxy` 反向代理组件平滑接管尚未迁移的管理后台与复杂用户聚合接口，实现混合架构下前后端无感联调。
- 实现 `AccessToken + RefreshToken` 双令牌鉴权方案，结合 Redis `tokenVersion` 字段支持即时强制下线，在"无状态 JWT"和"可主动吊销"之间取得工程平衡。
- 基于 `Redis` 构建互动读模型，承接点赞、收藏、评分等高频用户态状态查询，降低帖子详情和列表场景下对 MongoDB 的直接回源压力。
- 基于 `RabbitMQ` 重构异步事件链路，将帖子索引同步、评论副作用、用户资料冗余同步等操作从主业务链路中解耦，采用"主写成功 + 异步消费 + 最终一致性"的方式替代重型分布式事务。
- 设计并实现 `sync-sidecar`，消费业务事件完成 `MongoDB → Elasticsearch` 的增量同步与手动全量重建，通过回查 Mongo 当前状态判断写入或删除，以 Mongo 作为唯一数据源确保 ES 读模型的一致性。
- 拆分并落地 `comment-rpc / notification-rpc`，修复评论创建 MQ 失败未回滚、子评论删除 `replyCount` 重复扣减、一级评论软删误发 delete 事件、状态型通知重复写入等链路一致性问题。
- 为搜索建议词设计"原词优先但不盲目插入"策略：仅在 ES 有真实命中时将原词插入首位，避免建议框回显无搜索结果的词，提升搜索体验。
- 建设本地可观测性闭环，为 Go 服务接入 `OpenTelemetry`，为 Java 服务配置 Jaeger 导出，接入 `Loki + Promtail + Grafana + Prometheus`，统一 `service / traceId / requestId / routingKey` 日志字段，支持 HTTP、gRPC、MQ 三类链路联合排障。

## STAR 版项目复盘

### S: Situation

原始系统以 Java 服务为主，承担帖子主写、管理后台和大部分业务逻辑，但随着功能增加，开始暴露几个问题：

- 高频互动和搜索场景逐渐变成性能敏感点
- 评论、通知等能力边界不够清晰，服务职责偏重
- Mongo、Redis、ES、MQ 之间的异步链路缺乏统一治理
- 分散容器日志导致排障效率较低

### T: Task

目标不是推倒重来，而是在已有系统上完成渐进式演进：

- 用 Go 拆出高频、边界清晰、适合独立演进的领域服务
- 保持外部接口统一，降低前端联调成本
- 用 MQ 和 sidecar 打通主库与搜索库的同步链路
- 补齐日志、trace、统一响应结构和开发文档

### A: Action

#### 1. 网关平滑代理 + Go 领域服务拆分

基于 `go-zero` 实现了 `gateway-api / user-rpc / interaction-rpc / search-rpc / comment-rpc / notification-rpc` 等服务。

关键设计：网关内部封装了 `javaproxy` 组件，负责将鉴权解析出的用户身份（userId / role / nickname）注入为 HTTP header，并在透传时自动生成或接力 requestId / traceId，使 Java 侧和 Go 侧在日志和 trace 层面共享同一条链路。

#### 2. JWT + Redis tokenVersion 即时失效方案

标准 JWT 无法主动吊销。本项目采用"tokenVersion 自增"方案：登出或强制下线时，将 Redis 中该用户的 `tokenVersion` 自增，网关鉴权时将 JWT payload 中的版本号与 Redis 当前版本对比，旧 token 立即失效，不需要维护全量黑名单。

#### 3. 互动 Redis 读模型

针对"每次加载列表页都需要知道当前用户是否点赞"这类高频只读场景，在写入互动明细的同时同步更新 Redis Set 结构，读取时优先走 Redis，不回 Mongo。

#### 4. 评论与通知链路的一致性修复

在拆分过程中处理了一轮关键的一致性问题：

- 评论创建时 MQ 发布失败但接口错误返回成功 → 补充 Mongo 写入回滚
- 子评论删除时 `replyCount` 重复扣减 → 用 `$gt: 0` 条件更新防止负数
- 一级评论软删后仍误发 MQ delete 事件 → 软删分支跳过 MQ 发布
- 状态型通知重复写入 → 改用 upsert + 部分唯一索引，重复触发只更新不新增

#### 5. 可观测性闭环

为 Go 服务接入 `OpenTelemetry`，配置 Jaeger OTLP 导出；为 Java 服务补充 Jaeger HTTP 导出；接入 `Loki + Promtail + Grafana`，统一日志中的 `service / traceId / requestId / routingKey` 字段，使 HTTP、gRPC、MQ 三类链路可以通过 traceId 串联排障。

### R: Result

最终得到的是一套可以完整运行的混合微服务后端：

- 评论、通知、搜索、互动等模块具备独立服务边界
- 网关统一接入 Go RPC 与 Java 服务，前端无感知
- 发帖 → MQ → ES、评论 → 通知等链路具备最终一致性保障
- 本地环境支持日志检索、链路追踪和接口回归
- 各模块有 L1/L2 单元测试覆盖关键路径和一致性边界

## 面试高频问题与回答思路

### Q1：为什么要用 Go 和 Java 混合，而不是统一技术栈？

> Java 侧已有可运行业务核心，直接推翻重写风险高、收益低。Go 更适合拆出去承接高频、边界清晰、性能敏感的服务——比如评论、通知、搜索，这些模块接口稳定、职责单一，是最适合先迁的。双栈方案更贴近真实工程里的迁移场景，而不是理想化从零搭建。

### Q2：为什么搜索同步不直接在主业务里写 ES？

> ES 写入属于副作用，不应该阻塞主业务。搜索允许短时间延迟可见，用 MQ 异步消费比强事务更合理。更关键的是，sidecar 在消费事件时会回查 Mongo 当前状态，如果帖子在事件投递期间被删除，ES 侧也会跟着删，这比盲目写入更安全。

### Q3：为什么互动状态要用 Redis 读模型？

> 点赞/收藏状态是高频读，列表页每条帖子都要判断当前用户是否互动过。每次都回查 Mongo 延迟高、压力大。用 Redis 维护 Set 结构，写入时同步更新，读取时 O(1) 判断，把"N 次 Mongo 查询"变成"1 次 Redis pipeline"，延迟和吞吐都有明显改善。

### Q4：为什么评论和通知要独立拆成 RPC？

> 这两个模块边界天然清晰，调用频率高，又容易产生独立的副作用链路（评论触发通知，通知触发未读数变更）。拆出去之后能独立扩展、独立排障，也更容易在服务边界上做权限控制和接口版本管理。

### Q5：你在这个项目里处理过什么比较真实的 bug？

推荐重点讲评论一致性问题：

> 印象最深的是评论创建时 MQ 发布失败的问题。评论写入 Mongo 成功后，如果 MQ 发布失败，接口原来是静默返回成功的——这会导致帖子评论数没有增加，通知也不会触发，但评论数据已经存在，是个孤岛状态。后来我在 MQ 发布失败时同步执行 Mongo 删除回滚，并把错误上抛给调用方，调用方重试可以触发完整流程。

### Q6：JWT + Redis tokenVersion 怎么做到即时失效的？

> 标准 JWT 是无状态的，签发后无法撤销，只能等它自然过期。我用了一个轻量方案：Redis 里为每个用户维护一个 `tokenVersion` 计数器，JWT payload 里也嵌了这个版本号。网关鉴权时，把 JWT 中的版本号和 Redis 当前版本对比，如果不一致就拒绝。强制下线只需要把 Redis 里的版本号自增，所有旧 token 立即失效，不需要维护全量黑名单，也不需要每次都查完整的 token 记录。

## 如果面试官追问"你最能体现 Go 能力的部分是什么"

> 我觉得有两块。第一是把评论、通知、搜索这些模块从 Java 服务里拆出来，用 go-zero 跑起来，同时处理好拆分过程中暴露的一致性问题，比如 MQ 回滚、软硬删分支、状态型通知去重。第二是把 Redis 读模型、RabbitMQ 事件驱动、OpenTelemetry 链路追踪这些基础设施串起来，让系统不只是能跑，而是能在 Grafana 里通过 traceId 定位跨服务问题。

## 一、简历项目经历写法

### 版本一：适合简历直接落地

**Sharely / 分享派后端重构项目**

- 基于 `Go + go-zero + Spring Boot + RabbitMQ + Redis + MongoDB + Elasticsearch` 设计并实现混合微服务后端，将账号、互动、搜索、评论、通知等高频模块逐步从 Java 主服务中拆分为 Go RPC 服务，降低单体耦合并提升可演进性。
- 使用 `gateway-api` 统一承接 HTTP 请求，完成 JWT 鉴权、弱登录态兼容、跨服务数据聚合与协议整形，将帖子搜索、互动状态、评论列表、通知未读数等多源数据收敛为前端可直接消费的接口结果。
- 使用 `Redis Set / Hash + Dummy 占位 + Bloom Filter + 预热锁` 构建互动读模型，解决帖子点赞、收藏、评分等高频场景下的缓存穿透、击穿和冷启动预热问题，减少对 Mongo 的直接压力。
- 使用 `RabbitMQ + sidecar` 实现帖子和用户冗余信息到 Elasticsearch 的异步同步，避免在主写链路中引入分布式事务，并提供手动触发的 Mongo -> ES 全量重建接口，提升搜索索引修复能力。
- 拆分并实现 `comment-rpc / notification-rpc`，将评论读写和通知读写从 Java 中抽离；针对评论创建与删除链路补齐 MQ 发布失败回滚、二级评论计数修正、状态型通知去重 upsert 等机制，提升系统一致性和健壮性。
- 接入 `OpenTelemetry + Jaeger + Loki + Promtail + Grafana + Prometheus`，统一服务日志字段和 trace 透传，支持在本地环境快速追踪 HTTP、gRPC、MQ 三类链路问题。

## 二、建议的口头自我介绍版本

我做了一个内容社区后端项目，原本以 Java 为主，后来我把账号、互动、搜索、评论、通知这些高频模块逐步拆成了 Go 微服务。整个系统通过 Go 网关统一接入，请求内部会走 gRPC 调用或者代理到 Java 主业务服务。数据层上我用 MySQL 存用户主数据，用 Mongo 存帖子、评论和通知，用 Redis 做会话控制和互动读模型，用 RabbitMQ 做异步解耦，用 Elasticsearch 做搜索。这个项目里我重点做了评论/通知 RPC 拆分、Redis 互动读模型、Mongo 到 ES 的异步同步，以及 OpenTelemetry、Jaeger、Loki 这套本地观测链路。

## 三、面试 Q&A

### 1. 为什么你的项目没有直接全量迁到 Go，而是保留了 Java 主业务服务？

回答思路：

- 这是一个渐进式重构，而不是从零开始的全新项目
- Java 侧已经有比较完整的帖子主写、后台审核、关注关系和运营能力
- 我优先拆的是高频、边界清楚、并发敏感的模块，比如网关、账号、互动、搜索、评论、通知
- 这种拆法的价值在于风险更低，也更贴近真实业务迁移

可补一句：

“如果一次性重写所有域，短期内功能很容易失稳；而渐进式拆分可以先把收益最高的模块独立出来。”

### 2. 你的评论创建为什么要在 MQ 失败时做回滚？

回答思路：

- 评论创建不是只有插入一条评论记录那么简单
- 下游还有帖子评论数、通知生成、AI 回复等副作用依赖 `comment.create`
- 如果 Mongo 插入成功但 MQ 没发出去，用户表面上看评论成功了，但系统其它地方永远不会同步
- 所以我把它改成“主写 + 事件”统一成功才算成功
- 如果 MQ 发布失败，就删除刚插入的评论，并回滚根评论 `replyCount`

关键词：

- 最终一致性
- 异步副作用
- 半成功状态
- 补偿回滚

### 3. 互动服务里为什么要同时用 Dummy、Bloom Filter 和预热锁？

回答思路：

- 这是针对缓存读模型的三层保护
- Dummy 用来表示“这个 key 已经查过，确实没有真实数据”，防止空值反复穿透
- Bloom Filter 用来快速判断某个帖子或评论是否“可能存在互动数据”，减少不必要的 Mongo 预热
- 预热锁用来避免并发场景下多个请求一起回源，把 Mongo 打爆

总结一句：

“这三个机制分别解决空值穿透、无效预热和并发击穿，不是重复设计，而是对应不同问题。”

### 4. 为什么搜索排序要单独做一个 `hot` 策略，而不是直接按点赞数排？

回答思路：

- 单纯按点赞数排序会导致老帖长期霸榜
- 单纯按时间排序又会让新帖质量不分青红皂白地冲到前面
- 所以我在 ES 里用 `function_score`
  - 一部分来自点赞数
  - 一部分来自时间衰减
- 这样热度和时效性可以同时兼顾

可补一句：

“这个设计本质上是在做一个简化版推荐流排序，而不是简单数据库排序。”

### 5. 你的项目是怎么处理分布式事务问题的？

回答思路：

- 我没有引入 2PC 或 TCC
- 因为这个项目大多数跨服务动作本质上是主写 + 冗余更新，不值得上重事务方案
- 采取的是 RabbitMQ 异步解耦 + 幂等消费 + 必要补偿的方式
- 例如用户资料更新通过 `user.update` 异步刷新 ES 冗余字段
- 帖子索引损坏时还可以通过 sidecar 的全量重建接口补齐

总结句：

“核心思想是把强一致只留在主存储，跨服务冗余同步通过事件和补偿来做。”

## 四、面试官可能继续深挖的点

### Redis 为什么不用单纯缓存数据库结果？

可答：

因为互动场景本身就是高频状态读写，直接把点赞/收藏/评分建成 Redis 读模型，比“查 Mongo 再缓存”更直接，读写路径也更可控。

### 为什么通知要分普通通知和状态型通知？

可答：

评论、回复这种是事件历史，需要逐条保留；点赞、收藏、关注这种更像状态变化，如果不去重，通知中心会刷屏。

### Jaeger 和 Loki 在你项目里的作用分别是什么？

可答：

- Jaeger 看一条请求跨 HTTP、gRPC 的耗时链路
- Loki 看具体日志内容和上下文
- 两者结合可以做到“先看 trace 找到问题链路，再去日志里看具体参数和错误”
